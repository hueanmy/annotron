Tech Design — Realtime Presence Service

Epic ID: CF-517   Author: Platform Team   Status: Draft   Updated: Jul 18, 2026

1. Summary

Introduce a dedicated presence service so clients can see who is currently viewing or editing a shoot in realtime. Today presence is inferred from polling the review API every 10 seconds, which is both stale and expensive at scale.

Goal: sub-second presence updates for up to 500 concurrent editors per workspace, at a fraction of the current API load.

2. Architecture

A stateless gateway terminates WebSocket connections and forwards presence events to a Redis-backed fan-out layer. Presence state is ephemeral and never persisted to the primary database.

client ──ws──▶ presence-gateway ──▶ Redis Pub/Sub ──▶ presence-gateway ──ws──▶ other clients
                     │
                     └──▶ heartbeat TTL (15s) evicts stale sessions

2.1 Components

ComponentResponsibilityScale unit
presence-gatewayWS termination, auth, fan-out1 pod / 5k connections
Redis clusterPub/Sub + presence TTL keys3 primaries / 3 replicas
presence-apiREST snapshot for cold loads2 pods

3. Data Model

Presence is a Redis hash keyed by workspace:{id}:presence, with one field per active session and a value carrying the user id, cursor location, and last-seen timestamp.

4. Non-Goals

  • Persisting presence history for analytics (handled separately by the events pipeline).
  • Cross-region replication — v1 is single-region only.

5. Open Questions

  1. Do we need per-field locking, or is soft presence enough for v1?
  2. What is the fallback when the WebSocket handshake fails behind strict corporate proxies?
